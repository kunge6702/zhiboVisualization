# 本地数据：知情失败 + 手动导出导入逃生通道

工具的本地存储层（localStorage）在数据丢失或损坏时必须让用户知情，不得静默重置；同时提供手动导出/导入项目 JSON 的能力，作为跨设备迁移和灾难恢复的逃生通道。不引入 IndexedDB 或云同步。

**选择的理由：**

工具的核心价值是"到现场别翻车"。当前 `loadProjects` 与持久化 `useEffect` 中的 `catch(e){}` 完全静默——数据损坏时直接重置成示例项目，用户毫无感知；写入超 quota 时静默失败，用户以为已存。这是翻车的隐形源头：用户以为规划已保存，到现场打开发现是空的。

"知情"是底线：发生丢失/损坏时必须明确告知用户发生了什么，而不是用一份示例项目掩盖。知情之后，用户至少知道要找备份，而非在虚假的安全感里继续操作。

"导出/导入"是逃生通道，也是离线工作流的一等公民。ADR-0001 要求 PWA 离线，而离线场景下用户最核心的流程是"在办公室规划好、带到现场执行"——没有导出/导入，这个流程根本跑不通。导出/导入是纯本地操作，不依赖网络，与 PWA 离线完全兼容，且不阻塞未来云同步：未来接入 Supabase 时，导入的 JSON 可以作为初次同步的数据源。

存储技术仍用 localStorage。不引入 IndexedDB、不引入自动版本快照、不引入云同步——这些是实现选型，应在性能或容量真正撞墙时再单独决策，不在此 ADR 范围内。本 ADR 只定产品语义：对失败的态度 + 逃生口的存在。

**否决的方案：**

1. 仅改静默吞错为提示，不加导出/导入。等于把"静默丢"升级为"告知丢"——丢了还是丢了，没有恢复手段，跨设备迁移仍无解。
2. 现在上 IndexedDB + 自动备份 + 云同步。存储层整个重写，且云同步依赖 ADR-0002 的 Next.js + Supabase 迁移，时机未到。把实现选型与产品语义混在一起决策，违反决策粒度原则。

**约束条件：**

- 知情不等于阻断。损坏提示不得阻塞用户继续使用工具——告知后仍创建空白项目让用户操作，只是把"这是空白项目、不是你的原有数据"说清楚。
- 导出格式为人类可读的 JSON，包含单个项目的完整状态（设备、连线、需求），文件名含项目名与日期，便于现场逐个识别。
- 导入须校验 schema 与 typeId 有效性（防止导入旧版或损坏数据导致渲染崩溃），校验失败时明确提示原因，不静默丢弃。
